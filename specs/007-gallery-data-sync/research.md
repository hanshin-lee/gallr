# Research: Gallery Data Sync from Google Sheets (007)

**Feature Branch**: `007-gallery-data-sync`
**Generated**: 2026-03-20

## Decision 1: KMP app data layer — supabase-kt library vs. direct Ktor HTTP client

**Decision**: Use the **existing Ktor HTTP client** (`ExhibitionApiClient`) to call Supabase's PostgREST REST API directly. Do NOT introduce the `supabase-kt` library.

**Rationale**: The project already has Ktor 2.3.12 with `ktor-client-core`, `ktor-client-content-negotiation`, `ktor-serialization-kotlinx-json`, `ktor-client-okhttp` (Android), and `ktor-client-darwin` (iOS) fully wired in `shared/build.gradle.kts`. It also has `ExhibitionApiClient`, `ExhibitionDto`, and `ExhibitionRepositoryImpl` already written and structurally correct. The supabase-kt 3.x library requires Ktor 3.x, which would require a breaking version upgrade. The supabase-kt library provides no functionality we need beyond what Ktor already gives us — Supabase's PostgREST is a standard REST API callable with any HTTP client by adding two headers (`apikey` and `Authorization`). Using the existing client requires **zero new dependencies** and **minimal code changes** — only the base URL and auth headers need to be configured.

**Alternatives considered**:
- `supabase-kt 3.4.1` with Ktor 3.x upgrade: Rejected — requires upgrading Ktor from 2.3.12 to 3.x (breaking changes); violates Principle III (YAGNI); adds a transitive dependency chain for no net benefit over direct REST calls.
- `supabase-kt 2.x` (Ktor 2.x compatible): Rejected — supabase-kt 2.x is in maintenance mode; using a stale major version creates future migration debt.
- Direct Ktor + no ExhibitionApiClient refactor: The existing `ExhibitionApiClient` class is already the correct abstraction; it only needs its base URL and auth headers updated.

## Decision 2: Sync pipeline architecture — Google Apps Script

**Decision**: Use **Google Apps Script (GAS)** with an `onEdit` trigger (for near-real-time sync) plus a time-based backup trigger (every 5 minutes) that reads all rows from the Google Sheet and calls Supabase's PostgREST REST API to perform a full upsert.

**Rationale**: Google Apps Script runs inside Google's infrastructure with full access to the sheet owner's Google credentials — no separate OAuth flow or service account key needed to read the sheet. `UrlFetchApp.fetch()` in GAS can make HTTPS calls to any external URL, including Supabase. The script can be deployed, maintained, and edited entirely in the browser (Google Apps Script IDE). For a solo developer maintaining a small dataset (≤500 rows), this is zero-infrastructure: no server, no container, no cron daemon. The Apps Script quotas are generous for this use case (6-minute execution limit; 20,000 URL fetches/day on free tier; 90 min/day total runtime).

**Alternatives considered**:
- Supabase Edge Function + pg_cron polling Google Sheets API: Rejected — requires Google Sheets service account JSON key stored as a Supabase secret, plus setting up pg_cron on the Supabase Postgres instance. More moving parts than GAS for the same outcome.
- Vercel/Cloud Run cron job: Rejected — requires deploying and hosting a separate server-side service; over-engineered for a single-sheet sync.
- Make.com / Zapier automation: Rejected — introduces a paid third-party dependency for something achievable with free tools.

## Decision 3: Full replace vs. incremental upsert strategy

**Decision**: **Full replace on each sync** — the Apps Script deletes all rows from the Supabase `exhibitions` table, then inserts the current sheet contents as a fresh batch.

**Rationale**: The dataset is small (≤500 rows). A full replace is simpler to implement, avoids tracking row identity for deletions, and guarantees the database is always exactly in sync with the sheet. With upsert-only, a curator-deleted row would remain in the database unless the script tracked IDs. Row IDs in the sheet can be generated by the sync script (using a hash of name + venue + opening date) rather than requiring the curator to manage them. The full replace happens in a single SQL `TRUNCATE + INSERT` wrapped in a transaction, taking under 100ms for 500 rows.

**Alternatives considered**:
- Upsert (INSERT … ON CONFLICT DO UPDATE): Rejected — requires the curator to assign and maintain stable row IDs; deletes are not handled automatically.
- Change detection (compare sheet snapshot to DB): Rejected — requires storing a previous snapshot; significantly more complex for marginal benefit at this dataset size.

## Decision 4: Row ID strategy — script-generated hash

**Decision**: The Apps Script generates a stable `id` for each exhibition row by hashing `exhibition_name + venue_name + opening_date` (e.g. a short SHA-256 or a sanitised slug). This ID is used as the Postgres primary key. The curator does not manage IDs.

**Rationale**: The curator should not need to understand or maintain database identifiers. The spec states the spreadsheet interface must require no technical knowledge. A deterministic hash from stable fields (name + venue + opening date) produces a consistent ID even across full-replace cycles. It also allows the KMP app to bookmark exhibitions by ID persistently (bookmarks survive a full replace as long as the exhibition name, venue, and date don't change). If those fields change, a new ID is generated and the old bookmark silently becomes invalid — acceptable for a v1.

**Alternatives considered**:
- Auto-increment Postgres integer ID: Rejected — IDs change on every full replace, breaking bookmarks.
- UUID generated by the script per run: Rejected — same issue as auto-increment; new UUID on every sync breaks bookmarks.
- Curator-managed ID column in the sheet: Rejected — violates the "no technical knowledge required" constraint (FR-006).

## Decision 5: Supabase table access control — anon key for read, service role for write

**Decision**: Enable **Row Level Security (RLS)** on the `exhibitions` table with a `SELECT` policy for `anon` (public read). The Apps Script uses the **service role key** for write operations (upsert/truncate). The KMP app uses the **anon key** for read-only queries.

**Rationale**: This is the standard Supabase access pattern. The anon key is safe to bundle in the KMP app because the RLS policy only permits SELECT — no INSERT, UPDATE, or DELETE is possible via the anon key. The service role key (which bypasses RLS) is stored only in the Apps Script's script properties (server-side secret storage), never in the mobile app binary.

**Alternatives considered**:
- No RLS, rely on anon key restrictions at the API level: Rejected — without RLS, the anon key could be used to write data if Supabase defaults change; defence-in-depth requires RLS.
- Custom JWT auth for the sync script: Rejected — service role key is sufficient and standard for server-side sync jobs.

## Decision 6: Supabase URL and anon key storage in the KMP app

**Decision**: Store the Supabase project URL and anon key as **build-time constants** in `BuildConfig` (Android) and a Swift `Info.plist` constant (iOS), injected via environment variables during CI/CD. For local development, they are hardcoded in a gitignored `local.properties` / `LocalConfig.kt` file.

**Rationale**: The anon key is a publishable key (read-only due to RLS) — it is safe to bundle in the app binary, similar to an API key for a read-only endpoint. Supabase recommends this pattern explicitly. The project URL is not secret. This approach avoids runtime configuration complexity (no network call to fetch config) and is consistent with how other public API keys (e.g. Naver Maps) are handled in this project.

**Alternatives considered**:
- Remote config service: Rejected — adds infrastructure complexity for a value that rarely changes and is not sensitive.
- Gradle flavors / product flavors: Considered — would allow staging vs. production environments; deferred to a future feature.

## Decision 7: Google Sheet column structure

**Decision**: The sync script reads a Google Sheet with **exactly these columns in order** (row 1 = header):

| Column | Field | Required | Notes |
|--------|-------|----------|-------|
| A | `name` | Yes | Exhibition title |
| B | `venue_name` | Yes | Gallery/venue name |
| C | `city` | Yes | City |
| D | `region` | Yes | Region/district |
| E | `opening_date` | Yes | ISO date (YYYY-MM-DD) or Korean date (YYYY.MM.DD) |
| F | `closing_date` | Yes | Same format as opening_date |
| G | `is_featured` | No | `TRUE`/`FALSE` or `1`/`0`; defaults to FALSE |
| H | `is_editors_pick` | No | Same as above |
| I | `latitude` | No | Decimal degrees, blank = null |
| J | `longitude` | No | Decimal degrees, blank = null |
| K | `description` | No | Free text, may be empty |
| L | `cover_image_url` | No | HTTPS URL or blank |

**Rationale**: Explicit column ordering is simpler than named-header lookup for a GAS script and ensures the curator template is never ambiguous. Required fields (A–F) are the minimum for the exhibition to be valid. The script skips rows where any of A–F are blank and logs a warning. Date formats follow both ISO and Korean conventions to reduce curator error.

**Alternatives considered**:
- Named header lookup (find column by header text): Considered — more resilient to reordering; deferred as a v2 improvement if curators find the fixed-order constraint limiting.
